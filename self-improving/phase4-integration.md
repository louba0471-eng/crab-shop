# Phase 4 — 综合集成体系

> 长期建设。基于钱学森"综合集成"方法论：跨层次、跨领域、跨任务的整合。

---

## 目标

1. **知识模块化**：建立从原始记录到高层智慧的层次体系
2. **任务分解引擎**：复杂任务 → 子任务树 → 智能调度 → 结果整合
3. **跨域协同**：多个领域知识联合使用时的最优路径

---

## 知识层次体系（自下而上）

```
Layer 1 — 原始记录
├── self-improving/execution-log.md      （每次任务的原始过程）
├── self-improving/exploration-log.md    （探索过程原始记录）
└── memory/YYYY-MM-DD.md                 （每日日志 raw）
        ↓ 提炼
Layer 2 — 领域模式
├── self-improving/domains/feishu.md     （飞书领域）
├── self-improving/domains/web.md        （搜索领域）
├── self-improving/domains/code.md       （代码领域）
├── self-improving/domains/system.md     （系统领域）
└── self-improving/domains/enterprise.md （企业领域）
        ↓ 沉淀
Layer 3 — 高层智慧
├── MEMORY.md                            （长期记忆：偏好/原则/重要决策）
├── SOUL.md                              （身份定位：角色/风格/规则）
└── self-improving/memory.md             （自我偏好：模式/教训/规则）
```

**层级流转规则：**
- 执行成功 → Layer 1 原始记录 → **提炼** → Layer 2 领域模式
- 同一领域出现 3+ 次成功模式 → **升华** → Layer 3 相关章节
- 用户纠正/重大决策 → 直接写入 Layer 3 + 标注来源

## 知识模块间引用规范

当一个任务涉及多个领域时，THINK 阶段必须**显式引用**：

```
例：用户要求"用飞书日历约会议，然后把内容写成文档"

THINK 决策前引用：
  → domains/feishu.md       （日历部分）
  → domains/doc.md          （文档部分）
  → execution-log 最近记录  （历史是否做过类似任务）

综合结论：
  → 飞书约会议：查主日历 → 创建日程 → 添参会人
  → 飞书写文档：create-doc API → 填入日程信息
  → 合并最优路径：先日历确认时间，再文档起草内容
```

## 任务分解引擎（Task Decomposition）

### 触发条件

满足任一即为"复杂任务"，需触发分解：
- 涉及 **2+ 不同领域**（如飞书+代码）
- 包含 **3+ 个步骤**的线性流程
- 用户说"帮我做XXX"但**没有具体步骤**

### 分解原则

```
分解检查清单：
□ 任务目标是否清晰？
□ 子任务之间是串行还是并行？
□ 哪些子任务可以同时执行（并行）？
□ 哪些子任务依赖其他子任务的结果（串行）？
□ 是否有循环/重复结构？
□ 每个子任务的置信度是否 ≥ 0.7？
□ 子任务是否有失败风险，如何降级？
```

### 分解格式

```markdown
任务：[用户原始需求]

子任务树：
├─ 子任务A [并行] — 独立可执行
├─ 子任务B [串行，依赖A] — 等A完成
└─ 子任务C [并行] — 独立可执行，但依赖B的某个值

调度方案：
  Step1: A + C 并行执行
  Step2: B 执行（等Step1完成）
  Step3: 整合A+B+C结果 → 最终输出

风险点：
  - B失败 → 降级：使用A+C的缓存数据
  - C失败 → 降级：跳过C，手动补
```

## 子任务调度规范

### 并行判断

满足以下任一，子任务可并行：
- 无数据依赖
- 读操作（只查不写）
- 不同工具/领域

### 串行判断

满足以下任一，子任务必须串行：
- 输出是另一个的输入
- 写操作（修改同一文件/资源）
- 有因果顺序

### 并行调度执行

```
当有 N 个并行子任务：
  → sessions_spawn 每个子任务为独立 subagent
  → 使用 mode="run"（单次执行）
  → 各 subagent 结果汇总 → 整合 → 最终输出
  → 写入 execution-log（整体作为一次任务记录）
```

### 结果整合

```
整合检查：
□ 所有子任务都成功？ → 直接合并
□ 部分成功？ → 成功结果合并，失败部分标记"降级/手动"
□ 全部失败？ → 汇总失败原因，写入 recent_errors，交用户
```
