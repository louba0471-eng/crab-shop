# 通用领域经验（General）

> 跨领域的通用模式和任务分解模板

## 复杂任务分解模板

当遇到复杂任务时，按以下结构分解：

```markdown
## 任务：[描述]

### 目标确认
<用户最终要什么？>

### 子任务分解
1. [子任务A] — [并行/串行，依赖什么]
2. [子任务B] — [并行/串行，依赖什么]
3. [子任务C] — [并行/串行，依赖什么]

### 调度方案
Step1: [并行任务组]
Step2: [依赖Step1的任务]
...

### 风险控制
- 子任务X失败 → [降级方案]
```

## 通用降级策略

| 场景 | 降级方案 |
|------|---------|
| 工具调用失败 | 换同类工具（如 search_engine A → B） |
| API 超时 | 重试1次，仍失败 → 降级到本地知识 |
| 依赖资源不可用 | 跳过该步骤，标记手动补 |
| 置信度不足 | 不执行，记录 pending_review |

## 多域任务处理

当任务横跨多个领域时：
1. 先明确各领域边界
2. 识别数据依赖关系
3. 制定串行/并行调度方案
4. 合并时注意格式兼容

---

## 经验教训

1. **分解是手段不是目的**：3步能完成的不拆5步
2. **并行节省时间**：无依赖的步骤一定要并行执行
3. **降级要提前想好**：不要等到失败了才想降级
